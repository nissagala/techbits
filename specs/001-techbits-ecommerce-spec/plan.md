# Implementation Plan: TechBits eCommerce Application

**Branch**: `001-techbits-ecommerce-spec` | **Date**: 2026-05-31 | **Spec**: [spec.md](spec.md)

**Input**: Feature specification from `specs/001-techbits-ecommerce-spec/spec.md`

## Summary

TechBits is a full Laravel 11 + MariaDB eCommerce web application for selling computer accessories
to customers in Sri Lanka. The application has three user roles (Guest, Customer, Admin), 41
server-rendered Blade screens, a simulated payment flow with Luhn validation, email OTP at every
login, and a complete admin back-office. It is an academic coursework project for a Master of
Information Security program and must be built with conventional standard Laravel implementation.

Technical approach: standard Laravel MVC — routes → controllers (with Form Request validation)
→ Eloquent models → Blade views. One Laravel project; no SPA; no API layer; no Breeze/Jetstream.

## Technical Context

**Language/Version**: PHP 8.4, Laravel 11 (latest stable via composer create-project)

**Primary Dependencies**: Laravel 11.x framework; MariaDB 10.6+; Laravel Mail facade (SMTP/Mailtrap
for dev); Laravel's built-in session auth (no Breeze/Jetstream/Sanctum)

**Storage**: MariaDB relational schema (see data-model.md); database-backed sessions
(`SESSION_DRIVER=database`); `storage/app/public/products/` for product images (symlinked via
`php artisan storage:link`)

**Testing**: PHPUnit (Laravel built-in); no automated test suite required by spec — manual testing
against acceptance scenarios in spec.md is sufficient for this academic project

**Target Platform**: Linux web server (Apache/Nginx + PHP-FPM 8.4); timezone Asia/Colombo

**Project Type**: Server-rendered web application — eCommerce storefront + admin back-office;
no SPA, no REST/GraphQL API, no native mobile apps

**Performance Goals**: Academic demo — no latency or throughput targets; stable under basic
interaction so security scanning can run (SCOPE.md §4.2: "No SLAs, uptime targets")

**Constraints**: Single seller; English only; LKR only; Sri Lanka delivery only; one simulated
payment method; standard Laravel conventions only; no WAF, no advanced rate limiting, no
third-party security libraries; no real payment gateway integration

**Scale/Scope**: 41 screens (S1–S30, A1–A11); ~40 seeded products across 10 categories;
1 seeded admin account; single-server demo deployment

**Structure Decision**: Single Laravel project at the repository root. All 41 screens are
server-rendered Blade views. Admin lives under `Admin/` controller namespace and `admin/` view
subdirectory within the same project, protected by `EnsureAdmin` middleware. Admin routes defined
in `routes/admin.php` and registered with the `/admin` prefix via `bootstrap/app.php`.

Product soft-deletes are used (not visible as "soft" to users — behave like hard deletes from
storefront perspective) to enforce the "SKU not reusable after deletion" business rule without a
separate deleted-SKUs tracking table. The `deleted_at` column causes the unique SKU constraint to
remain effective for deleted records.
